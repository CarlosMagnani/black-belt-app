Solo Dev Launch Analysis · 2026-05-19

BlackBelt
Priority Plan

The v0.1.0 roadmap lists ~40 items — that's 3–4 months solo. This plan defines the smallest slice that gets a real academy using the product.

The Non-Negotiable Core Loop

These 7 capabilities must exist before the product has any value. Nothing else ships until this loop works end-to-end.

Owner creates academy → invite code generated
→ Student enters code → joins academy
→ Owner creates class schedule
→ Student sees today's classes → requests check-in
→ Owner approves check-in
→ Counter increments → belt progress updates
→ Owner promotes student manually

What to Cut from v0.1.0

These items are in the current roadmap and should be deferred to v0.2.0.

Item Why Cut
Membership Plans (entire section) No payment = no user value. Admin paperwork with zero functional benefit for the first academy.
Professor management Adds a full permissions surface. Owner-only v1. Owner can teach classes too.
Password reset Use Supabase dashboard for early users. Manual is fine for 5 academies.
Owner Dashboard — Revenue Card Requires manual subscription data nobody will enter. No payment processing anyway.

Student — Streak / heat strip UI-heavy. Replace with "X classes this week."
Student — Monthly stats card Same as above. No data history to fill it at launch.

Phase Plan

00 Scaffold 2–3 days

One-time setup everything else runs on. Do it properly — don't come back to it.

  • Init frontend: Vite + React + TypeScript + Tailwind (with BlackBelt tokens in tailwind.config.ts)
  • Init backend: Fastify + TypeScript + Prisma + Zod
  • Deploy empty shell to Railway — confirm env vars and Postgres connection work
  • Apply full Prisma schema (all 8 models) — build it now, use only what you need
  • Set up folder structure per architecture doc
Critical spike
  • Supabase Auth + Fastify JWT — spike in isolation before building features on top. Confirm SUPABASE_JWT_SECRET validates correctly via @fastify/jwt. If it takes more than 1 day, evaluate switching to simple JWT-in-Postgres.
01 Auth + Onboarding 5–7 days

Gets users into the product with the right role.

Backend
  • POST /auth/register, POST /auth/login, POST /auth/logout
  • Auth middleware — JWT validation on all protected routes
  • POST /academy — create academy, generate invite code, enforce one-per-owner
  • POST /academy/join — validate invite code, create AcademyMember
  • GET /academy/:id/invite — return invite code for owner
Frontend
  • ScreenSplash — role split (Master vs Aluno)
  • ScreenOwner — 3-step onboarding (academy → profile + belt picker → invite code reveal)
  • ScreenStudent — 3-step onboarding (code entry + verify → profile → belt picker)
  • BeltPicker widget + BeltVisual widget
  • Route guards — redirect unauthenticated users
02 Class Schedule 4–5 days

Owner builds the academy's weekly schedule. Students see it.

Backend
  • POST /schedule — create class (day, time, duration, instructor, location, level)
  • PUT /schedule/:id — edit class
  • PATCH /schedule/:id/deactivate — soft delete
  • GET /schedule — list active classes for the academy
Frontend
  • Owner: class creation form + weekly schedule view
  • Student: today's classes list + weekly view
  • ClassRow component (time block + title + professor + level)
03 Check-in Loop 4–5 days

The core value exchange. Students attend, owners confirm. This is the product.

Backend
  • POST /checkin — student requests (status starts pending, one per class per date)
  • GET /checkin/pending — owner sees pending list
  • PATCH /checkin/:id/approve — triggers belt progression check
  • PATCH /checkin/:id/reject
  • Domain rule enforced server-side: student cannot approve their own check-in
Frontend
  • Student: ScreenClass — class detail + check-in button (simple button in v1)
  • Owner: pending check-in list with approve / reject actions
04 Belt Progression 3–4 days

Turns check-ins into meaningful progress. The domain logic that makes BJJ management real.

Backend
  • On check-in approval: increment approved_classes_at_level
  • If count ≥ 24: advance degree (or belt if degree = 4), reset counter
  • POST /belt/promote — owner manually promotes, creates BeltProgressionEvent
  • GET /students/:id/belt — current belt, degree, progress count
Domain rules (pure functions, tested)
  • isBeltAdvancementEligible(approvedClasses) → boolean
  • calculateNextRank(belt, degree) → { belt, degree }
  • isCheckInSelfApproval(studentId, approverId) → boolean
Frontend
  • Student home: belt progress bar (X/24 to next degree) + current belt visual
  • Owner: promotion queue + promote button
05 Minimal Dashboards 3–4 days

The home screens after onboarding. Stripped to what's actionable on day one.

Student home
  • Today's date + greeting
  • Belt progress bar
  • Today's classes
  • Recent check-in status (approved / pending / rejected)
Owner home
  • Today's schedule
  • Pending check-ins count (the action item)
  • Student count by belt — roster snapshot (single query, worth it)
  • Promotion queue — students ready to advance

Technical Risks

Supabase Auth + Fastify JWT

The handoff flags this as "not plug-and-play." Spike it in Phase 0 before building anything on top. If it takes more than 1 day, evaluate a simple jsonwebtoken + custom middleware — at MVP scale it's equivalent.

Tailwind + Design Tokens

Do this once in Phase 0. Extend the theme with BlackBelt tokens, disable Tailwind defaults. If Tailwind defaults creep in (slate-700, red-500) the design breaks and is hard to audit later.

Prisma + Railway Postgres

Railway's managed Postgres uses a simple single connection URL — no pooler complexity unlike Supabase Postgres. Confirm DATABASE_URL works with a single connection string before configuring anything else.

Design Polish — Deferred to v0.2.0

The design handoff is high-fidelity and final. But for a solo developer, the interaction model matters more than visual polish in v1. These are additive — they don't change any data model or API.
